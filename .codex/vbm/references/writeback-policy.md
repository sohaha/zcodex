# 回写策略

只有“值得下次继续用”的知识，才应该被写回 `.ai`。

## 应该写回的内容

- 配置位置和配置优先级
- 稳定业务规则
- 已验证的问题根因
- 容易被改坏的回归风险
- 已落地且影响后续实现的技术决策

## 不应该写回的内容

- 临时猜测
- 未验证结论
- 一次性调试日志
- 可从代码直接明显看出的废话
- 密码、token、私钥、完整连接串等敏感数据

## 写回原则

1. 先验证，再落盘。
2. 同类记忆只保留当前有效结论，不制造冲突版本。
3. 能写进 `project/` 的稳定事实，不要散落到 `bugs/` 或 `decisions/`。
4. 问题记录和决策记录的标题要让下一次能快速检索到。
