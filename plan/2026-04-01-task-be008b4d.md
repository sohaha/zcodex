# 分析 codex 因 tldr semantic ONNX Runtime 动态库缺失导致 panic 的原因与后续方案

> 不适用项写 `无`，不要留空。
> 只写已确认事实；若信息不清楚或需要假设，先回到对话澄清，不要把未确认内容写进计划。

## 背景
- 当前状态：已确认 codex 在执行 `tldr` 工具的 `action=semantic` 时会崩溃，报错包含 `Failed to load ONNX Runtime dylib`、`libonnxruntime.so: dlopen failed` 与 `JoinError::Panic`。
- 触发原因：`codex-native-tldr` 通过 `fastembed` 使用启用了动态加载 ORT 的 `ort`；当前运行环境未设置 `ORT_DYLIB_PATH`，常见动态库搜索路径中也未发现 `libonnxruntime.so`。`ort` 在动态加载失败时直接 `panic!`，该 panic 发生在 tool task 中，最终被 codex 外层上报为 `in-flight tool future failed during drain`。
- 预期影响：需要把这次问题的事实边界、受影响链路、可选处置路径和后续验证方式固化，供后续 issue generation 和 execution 使用。

## 目标
- 目标结果：形成一份只基于已确认事实的计划，明确本问题的根因、影响范围、非目标、后续处置路径和验证入口。
- 完成定义（DoD）：
  - 计划中明确区分已确认根因、表层症状和受影响调用链；
  - 计划中明确区分环境/配置规避路径与代码级修复路径；
  - 计划中列出后续可执行的验证入口与风险点；
  - 计划不包含未确认假设、占位符或范围漂移。
- 非目标：
  - 不在本阶段直接修改代码；
  - 不在本阶段安装或分发 ONNX Runtime 动态库；
  - 不在本阶段决定最终采用环境修复还是代码修复；
  - 不扩展到与本次 panic 无关的 `tldr`、`fastembed` 或工具路由问题。

## 范围

- 范围外：具体代码实现、依赖升级、发布流程变更、与本次崩溃无关的其他 CLI/TUI 行为调整。

## 影响
- 受影响模块：`codex-rs/native-tldr`、`codex-rs/core`、`codex-rs/mcp-server`、`codex-rs/cli`
- 受影响接口/命令：`tldr` 工具的 `semantic` action；结构化查询首次触发时的自动 `Warm`；可能走到 `tldr semantic` 的 codex 代码理解流程
- 受影响数据/模式：运行时动态库加载路径；`ORT_DYLIB_PATH`；系统动态链接器搜索路径；`native-tldr` 默认 semantic 配置
- 受影响用户界面/行为：用户在 codex/TUI 中执行依赖 `tldr semantic` 的代码理解流程时，可能看到整个应用因 tool panic 崩溃

- 约束（时间/兼容性/安全/性能/发布窗口等）：遵守“先别直接改代码”；只记录已确认事实；后续若进入执行阶段，需要避免用静默 fallback 掩盖真实错误；如选择代码修复，应保持最小范围变更。
- 外部依赖（系统/人员/数据/权限等）：若采用环境修复路径，依赖运行环境提供可加载的 `libonnxruntime.so` 或显式配置 `ORT_DYLIB_PATH`；若采用代码修复路径，依赖仓库内后续 issue 与实现评审。

## 实施策略
- 总体方案：先把已确认事实和后续选项固化到计划，再在 issue generation 阶段把工作拆分为“环境/配置规避验证”和“代码级根因修复设计/实现”两类 issue，避免在执行阶段混淆问题定位与方案选择。
- 关键决策：
  - 把根因界定为 `tldr semantic -> native-tldr -> fastembed -> ort(load-dynamic)` 链路上的运行时动态库缺失与 panic 传播，而不是 daemon unavailable 或普通工具错误；
  - 把 `Warm` 视为伴随动作而非致命点，致命点是随后的 `action=Semantic`；
  - 在后续 issue 中显式保留两条路径：环境/配置规避路径与代码级修复路径，由用户后续选择或并行评估。
- 明确不采用的方案（如有）：
  - 不把当前问题归因为 `Warm` 失败；
  - 不把当前问题归因为单纯的 tool routing 配置；
  - 不在无确认情况下直接决定“禁用 semantic”就是最终产品方案；
  - 不在计划阶段写入未验证的第三方行为推测。

## 测试与验证
- 核心验证：基于源码、配置和本地日志，确认崩溃前存在 `ToolCall: tldr {"action":"semantic"...}`，并在 `tldr begin action=Semantic` 之后出现 `Failed to load ONNX Runtime dylib` 与 `JoinError::Panic`。
- 必过检查：确认默认 `native-tldr` 配置启用了 semantic/embedding；确认当前环境未设置 `ORT_DYLIB_PATH` 且未找到 `libonnxruntime.so`。
- 回归验证：后续进入执行阶段时，需要分别验证环境修复路径能否消除加载失败，以及代码修复路径能否把第三方 panic 收敛为显式错误而非整进程崩溃。
- 手动检查：查阅 `codex-tui.log` 中的 `ToolCall: tldr`、`tldr begin`、panic 与 drain 失败日志顺序；查阅 `Cargo.toml`、`README`、`config.rs`、`tool_api.rs`、`parallel.rs`、`codex.rs` 中相关配置和传播链路。
- 未执行的验证（如有）：本阶段未实际安装 ORT 动态库，也未运行带修复的行为验证，因为当前阶段只做计划且用户要求先别直接改代码。
