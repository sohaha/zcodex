# 优化 TUI 宠物功能的常驻占位

## 背景
- 当前状态：TUI 启动时只要 `tui_show_buddy` 为真，就会在 `ChatWidget` 初始化阶段调用 `ensure_buddy_visible`，让 buddy 直接进入可见状态；配置默认值目前也是开启。启动后，bottom pane 会在 buddy 可见时额外插入一个气泡 renderable 和一个宠物 renderable，因此在宽终端下会稳定占用多行高度；宠物反应气泡默认持续 10 秒。
- 触发原因：用户反馈“现在的宠物功能启动之后平时占用了大量空间”，并进一步明确期望：默认启动时先显示全形象，但过一会应自动不再维持全形象；若用户执行 `/buddy hide`，则之后应完全不显示；同时保留一个显式进入“全形象常驻”的命令。
- 预期影响：通过把“默认启动全形象”改成短暂开场态，再自动回落到更省空间的常驻形态，来保留宠物的存在感而不长期挤压 bottom pane；并继续支持用户显式隐藏或显式开启全形象常驻。

## 目标
- 目标结果：让 buddy 在默认启动时先以全形象短暂出现，随后自动回落到非全形象的常驻状态，避免长期占用多行空间；如果用户执行 `/buddy hide`，则后续完全不显示；同时在 `/buddy` 下保留一个显式进入“全形象常驻”的命令。
- 完成定义（DoD）：
  - 默认配置下启动 TUI 时，buddy 会先短暂显示全形象，然后自动退出全形象状态。
  - 用户执行 `/buddy hide` 后，buddy 会完全隐藏，并在后续启动中保持不显示，直到再次显式开启。
  - `/buddy` 新增或保留一个可显式开启“全形象常驻”的命令，并具备清晰、可持续的显示语义。
  - 现有 `/buddy show|pet|hide|status` 语义经过调整后仍保持可用，用户仍能获得明确反馈。
  - 受影响的 TUI 渲染测试与 snapshot 更新并通过。
- 非目标：
  - 不移除 buddy 功能本身。
  - 不改动 app-server、MCP 或 core 的协议面。
  - 不在本轮引入新的模型提示注入、人格逻辑或额外配置项，除非实施中发现现有边界无法满足目标。

## 范围
- 范围内：
  - 调整 `codex-rs/tui` 中 buddy 的启动展示策略、自动回落策略与渲染/展示策略。
  - 在 `/buddy` 下增加或保留“全形象常驻”命令，并定义它与现有 `show/hide/status/pet` 的关系。
  - 调整 bottom pane 对 buddy 气泡与主体的布局接线。
  - 更新相关快照与命令/渲染测试。
- 范围外：
  - buddy 灵魂生成、reaction 内容生成策略。
  - 非 TUI 运行面的任何行为变更。

## 约束与依赖
- 约束（时间/兼容性/安全/性能/发布窗口等）：
  - 需要遵守 `codex-rs/tui` 的快照测试要求，任何用户可见 UI 变化都要补 snapshot。
  - 需要保持改动局部化，优先复用现有紧凑渲染能力，避免把复杂逻辑继续堆到 `chatwidget.rs`。
  - 不能未验证就宣称优化完成；Rust 改动完成后要执行 `just fmt` 和 `codex-tui` 定向测试。
- 外部依赖（系统/人员/数据/权限等）：
  - 无新增外部依赖；只依赖现有 Rust/TUI 测试工具链。

## 实施策略
- 总体方案：
  - 保留当前默认可见配置，但把“启动即显示”收敛为“启动短暂全形象 + 自动回落”的双阶段展示，而不是长期停留在完整 sprite/bubble。
  - 在 `/buddy` 下新增或保留一个 `full` 命令，显式切换到长期显示完整 sprite/bubble 的模式；未开启该模式时，buddy 在开场阶段结束后自动回到更省空间的常驻形态。
  - `/buddy hide` 继续作为彻底关闭入口，并保持现有持久化语义；一旦用户关闭，后续启动不再自动显示。
  - 复用现有 buddy 渲染能力与状态结构，在尽量少改动命令分发的前提下，为“启动开场全形象 / 普通常驻 / 全形象常驻 / 完全隐藏”定义清晰边界。
- 关键决策：
  - 采用用户明确指定的产品决策：默认启动先显示全形象，但只持续短时间，随后自动退出全形象。
  - 用户通过 `/buddy hide` 关闭后，应完全不显示，而不是只退出全形象。
  - 新增命令默认方案采用 `/buddy full`，用于开启全形象常驻；除非实现中发现与现有命令体系存在实际冲突，否则不再引入额外命名分支。
  - 现有 `/buddy show` 默认保留为普通显示语义，不直接承担“全形象常驻”职责，避免与新增命令混淆。
  - `/buddy pet` 等显式交互仍应保留可感知反馈，但必须与“启动开场态”和“完全隐藏态”兼容。
- 明确不采用的方案（如有）：
  - 不把默认行为改成彻底不显示，因为这与最新产品语义冲突。
  - 不对 buddy observer / soul 生成链路做顺手扩展。
  - 不通过新增一组独立配置项把“启动开场态 / 全形象常驻 / 隐藏”实现为复杂的多开关矩阵；优先由现有显示开关 + 命令语义驱动。

## 阶段拆分
### 阶段一：收敛命令与显示语义
- 目标：明确“启动开场全形象”“普通常驻”“全形象常驻”“完全隐藏”四种状态的命令与运行态边界。
- 交付物：`/buddy` 子命令方案、显示模式定义、自动回落规则。
- 完成条件：工程上能够明确区分“启动开场全形象”“普通常驻”“全形象常驻”“完全隐藏”。
- 依赖：现有 `BuddyWidget`、`BuddyBubble`、`/buddy` 命令分发链路。

### 阶段二：实现启动开场与全形象常驻
- 目标：落地“启动短暂全形象 + 自动回落”，并新增用户可显式开启完整形态常驻的命令与状态。
- 交付物：命令分发调整、buddy 渲染/状态调整、自动回落时机处理、bottom pane 接线更新。
- 完成条件：启动后先出现全形象，随后稳定回落；执行新命令后能稳定进入全形象常驻模式；执行隐藏后彻底不显示。
- 依赖：阶段一的命令与状态规则。

### 阶段三：验证与快照更新
- 目标：补齐回归证据，确保 UI 改动可审阅、可复现。
- 交付物：更新后的 buddy snapshot、相关测试调整、格式化结果。
- 完成条件：定向测试通过，快照反映新的启动开场、自动回落、隐藏与全形象常驻行为。
- 依赖：前两阶段实现完成。

## 风险与缓解
- 关键风险：新增“启动开场全形象”后，若自动回落条件不清晰，容易造成持续占位、闪烁，或与 `/buddy full`、`/buddy hide` 语义冲突。
- 触发信号：启动后长时间不回落；回落后状态异常；`/buddy show` 与 `/buddy full` 行为边界不清；`/buddy hide` 后重启仍再次出现；状态栏/inline preview 被挤压或错位。
- 缓解措施：先明确四种显示状态与状态转移；把自动回落时机收敛到 buddy 渲染边界；补充启动开场、自动回落、显式 full、hide 持久化四类测试与 snapshot。
- 回滚/恢复方案（如需要）：若“启动开场全形象”接线引入过大回归，可先保留 `/buddy full` 与 `/buddy hide` 的命令语义，把默认开场态收缩为最小版本后继续迭代。
