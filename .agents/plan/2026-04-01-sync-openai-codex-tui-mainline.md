# 同步 openai/codex 上游并将 `codex-rs/tui` 升级为主 TUI 实现

## 背景
- 当前状态：当前分支为 `web`，同步基线记录在 `/workspace/.codex/skills/sync-openai-codex-pr/STATE.md`，上次上游基线为 `e590fad50b832bebc68e7f18067cad5fe13a7d85`；独立 worktree 已创建于 `/.worktrees/sync-openai-codex-20260401-104254`，并已尝试将 `openai/main`（当前目标 `dedd1c386a9bb2b4031d26b0494217a20868fb7a`）合入该 worktree。

## 目标
- 目标结果：在独立 worktree 中完成 `openai/main` 到当前仓库的同步，采用上游 `codex-rs/tui` 作为主 TUI 实现方向，移除仓库内将 `codex-rs/tui` 定义为“兼容层”的限制，并在合并过程中保留当前本地分支已有功能与用户可见行为。
- 完成定义（DoD）：
  - worktree 中完成上游同步、冲突解决与必要中文化；
  - `AGENTS.md`、`codex-rs/README.md` 等仓库说明不再把 `codex-rs/tui` 定义为兼容层；
  - 对本地现有 TUI/CLI 相关功能做“保留或等价迁移”核对，最终总结中明确说明未发现明显功能丢失/覆盖，或明确列出剩余风险；
  - 按仓库规则完成相关格式化与最小必要测试/快照验证；
  - 更新 `STATE.md`、合并回原分支、提交，并同步更新交接记忆。
- 非目标：
  - 无依据地删除与本次同步无关的本地定制功能；
  - 额外重构与本次同步无直接关系的模块；
  - 在未验证的情况下宣称“完全无风险”。

## 实施策略
- 总体方案：先在现有冲突 worktree 中做“本地功能盘点 + 上游落点映射”，把当前仓库里 `tui_app_server` 主实现承担的用户可见功能、测试与中文化差异整理为迁移/保留清单；再以 `codex-rs/tui` 为主线解决冲突，删除或调整把 `tui` 视为兼容层的仓库约束，最后通过定向测试、snapshot、文档与总结证明本地功能未被明显丢失。
- 关键决策：
  - 采用上游 `codex-rs/tui` 主实现方向，而不是继续保留 `tui_app_server` 为主；
  - 对“本地功能不丢失”的执行方式不是盲目保留所有旧文件，而是先核对本地现有行为，再把这些行为保留在新主实现中或给出等价迁移；
  - 先改真实源码与说明，再同步快照、断言与文档，避免只改测试掩盖实际行为偏差。
- 明确不采用的方案（如有）：继续把 `codex-rs/tui_app_server` 作为主实现并仅局部移植上游 TUI 改动的方案，本轮不采用。

## 测试与验证
- 核心验证：针对本轮实际改动的 crate 运行最小必要测试；若 TUI 相关变更落在 `codex-rs/tui`，优先运行对应 crate 测试与 snapshot 流程；若公共 crate（如 `core`）被改动，先完成定向测试，再按仓库规则决定是否需要请求完整测试。
- 必过检查：`just fmt`；受影响 crate 的编译/测试；若 snapshot 变化，执行 `cargo insta pending-snapshots -p codex-tui` 并审阅/接受预期变更；必要时执行相关 `just fix -p <project>`。

- 未执行的验证（如有）：若完整工作区测试或部分环境相关验证因成本/范围未执行，最终总结中明确列出。

## 风险与缓解
- 关键风险：
  - 上游将主实现迁到 `codex-rs/tui`，而本地当前仍以 `tui_app_server` 为主，存在本地定制能力在迁移中遗漏的风险；
  - 当前主工作区存在未提交改动，回合并时可能与本次同步改动互相干扰；
  - snapshot 与中文化差异较大，可能出现“测试通过但文案回退”或“文案已改但快照未同步”的不一致。
- 触发信号：
  - 冲突解决后，功能清单中存在未找到落点的本地能力；
  - 定向测试/快照出现与本地现有行为不符的回退；
  - 最终 diff 中仍保留把 `codex-rs/tui` 视为兼容层的文字或代码路径。
- 缓解措施：
  - 先建立功能保留清单再动手融合；
  - 对高风险行为使用定向测试、snapshot 与源码对照三重核对；
  - 在合回原分支前再次检查只包含本次同步相关改动；
  - 最终总结中显式列出仍无法完全证明的剩余风险。
- 回滚/恢复方案（如需要）：保留独立 worktree 作为隔离现场；若迁移方向导致无法在当前轮次内证明本地功能可保留，则在 worktree 内停止并回报，不直接污染主工作区；若已写入 `STATE.md` 但最终未落地提交，必须恢复为真实已同步基线。
