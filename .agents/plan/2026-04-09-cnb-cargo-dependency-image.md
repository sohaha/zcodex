# CNB Cargo 依赖镜像方案调研

## 背景
- 当前状态：当前任务运行在使用 CNB 的云开发环境中；根据用户反馈，每次进入环境都需要重新下载/安装 Cargo 依赖，耗时明显。
- 触发原因：用户希望依据 CNB 文档寻找“依赖中间层镜像/中间依赖镜像”方案，减少重复下载。
- 预期影响：形成可执行的技术方案比较与推荐，支撑后续在当前云开发环境中落地依赖缓存优化。

## 目标
- 目标结果：调研并输出适用于 CNB 场景的 Cargo 依赖复用方案，至少覆盖官方支持机制、适用前提、实施方式、限制与推荐顺序。
- 完成定义（DoD）：
  - 基于可追溯来源完成调研，优先使用官方 CNB 文档；
  - 明确至少 2 种可行路径，并说明差异；
  - 给出针对“云开发环境反复重建导致 Cargo 依赖重复下载”的推荐方案；
  - 输出后续若进入实施阶段需要补充确认的环境信息。
- 非目标：
  - 本阶段不直接修改仓库构建配置；
  - 不假设具体云厂商或具体 CNB 平台实现细节；
  - 不产出未经文档证据支撑的实现结论。

## 范围
- 范围内：
  - 调研 CNB 官方或主流实现文档中与依赖复用相关的机制；
  - 识别哪些机制适合缓存 Cargo 依赖、哪些更适合预置工具链/系统包；
  - 结合当前“环境频繁重建”的约束给出推荐顺序。
- 范围外：
  - 直接改造 builder、buildpack、Dockerfile 或 CI 配置；
  - 进入特定平台控制台验证；
  - 为未知平台编造兼容性承诺。

## 影响
- 受影响模块：无（本阶段为调研与方案输出）。
- 受影响接口/命令：可能涉及后续实施阶段的 `pack build`、builder/build image 配置、缓存镜像配置或 buildpack 配置入口。
- 受影响数据/模式：可能影响后续构建缓存、依赖下载源与镜像分层策略。
- 受影响用户界面/行为：无直接界面影响；预期影响后续构建耗时与网络依赖。

## 约束与依赖
- 约束（时间/兼容性/安全/性能/发布窗口等）：
  - 仅能基于公开资料与当前仓库上下文做结论；
  - 必须优先采用官方 CNB 机制，避免依赖脆弱私有 hack；
  - 结论需要区分“缓存构建产物/依赖层”和“预置基础镜像内容”两类方案。
- 外部依赖（系统/人员/数据/权限等）：
  - 需要访问 CNB 官方文档及必要的主流 buildpack 文档；
  - 后续实施前可能需要用户补充当前云开发环境所用 CNB 平台、builder/buildpack 发行版与镜像仓库权限。

## 实施策略
- 总体方案：
  - 先确认 CNB 官方支持的持久化/跨机器复用机制；
  - 再补充 builder/build image/自定义 buildpack 等“预置依赖”路径；
  - 最后按实施成本、适配风险、对 Cargo 依赖下载加速的直接性进行排序并给出推荐。
- 关键决策：
  - 优先将“cache image”作为官方首选候选，因为它直接面向临时环境中的跨机器构建层复用；
  - 将“自定义 build image / builder”视为备选，因为它更适合预装系统级依赖或工具链，而不天然等价于 Cargo registry/cache 复用；
  - 如发现主流 Rust buildpack 有明确的依赖镜像/镜像源配置能力，再纳入比较。
- 明确不采用的方案（如有）：
  - 不把普通 Docker layer cache 直接等同于所有 CNB 平台上的通用解法；
  - 不把未证实的供应商特性当作 CNB 标准能力。

## 阶段拆分
### 阶段一：官方机制核实
- 目标：确认 CNB 标准层面与依赖复用直接相关的机制。
- 交付物：官方机制清单、适用条件与限制说明。
- 完成条件：形成至少一项来自 CNB 官方文档的明确可行路径。
- 依赖：buildpacks.io 官方文档。

### 阶段二：Rust/Cargo 场景映射
- 目标：判断各机制如何映射到 Cargo 依赖缓存、registry mirror、工具链预置等实际问题。
- 交付物：方案对比表与适配分析。
- 完成条件：清楚区分“缓存 Cargo 依赖”和“预置构建环境”两类方案价值边界。
- 依赖：官方文档与主流 buildpack 资料。

### 阶段三：推荐结论输出
- 目标：给出推荐方案、前置条件与实施前待确认项。
- 交付物：面向当前云开发环境的建议清单。
- 完成条件：用户可据此决定下一步进入 issue 生成或实施。
- 依赖：前两阶段调研结论。

## 测试与验证
- 核心验证：核对每个结论是否能追溯到明确来源，并区分事实、推断与建议。
- 必过检查：至少一条官方 CNB 文档直接支持核心推荐；涉及限制时明确标注适用前提。
- 回归验证：无（本阶段不涉及代码或运行时行为变更）。
- 手动检查：复核方案是否直接回应“云开发环境重复下载 Cargo 依赖”的问题，而不是泛化成无关的镜像定制建议。
- 未执行的验证（如有）：未在具体云平台或具体 builder 上做实机验证；后续实施阶段需要补齐。

## 风险与缓解
- 关键风险：不同 CNB 平台对缓存镜像、registry 权限或 buildpack 行为支持程度不同。
- 触发信号：官方方案存在，但当前平台缺少 `--cache-image`、远端 registry 推送或自定义 builder 能力。
- 缓解措施：在结论中明确区分“标准机制”“平台前提”“若平台受限时的备选路径”。
- 回滚/恢复方案（如需要）：无（本阶段不落地修改）。
