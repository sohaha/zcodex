# CLI shared options 汉化必须落在 `utils/cli` 文案源头

## 背景
- 顶层 `codex`/TUI help 已经通过 `localize_help_output()` 把 `Usage`、`Options`、`Print help` 之类框架级文本翻成中文。
- 但 `--image`、`--model`、`--oss`、`--local-provider`、`--profile`、`--sandbox`、`--full-auto`、`--dangerously-bypass-approvals-and-sandbox`、`--cd`、`--add-dir` 这类共享参数的说明仍显示英文。

## 结论
- 这些正文不是由 `localize_help_output()` 生成，而是直接来自 `codex-rs/utils/cli/src/shared_options.rs` 的 clap help/doc 注释。
- 因此修复这类回归时，不能只在 `codex-rs/cli/src/main.rs` 做输出级字符串替换；必须回到 `SharedCliOptions` 的定义处更新文案源头。
- 同时要在 `codex-rs/cli/src/main.rs` 保留顶层 help 断言，既检查中文说明存在，也检查旧英文说明已经消失，避免只本地化标题和占位符、漏掉共享参数正文。

## 后续规则
- 只要 help 文案来自 `SharedCliOptions`，`cli`、`tui`、`resume/fork` 等复用入口都会继承同一份说明；优先改共享定义，不要在单个入口各自补丁。
- 调整帮助子命令本地化时，要顺手核对 `autogenerated_help_subcommand_is_localized` 这类断言与真实输出的句号/标点是否一致，避免无关断言漂移挡住 CLI 回归验证。
