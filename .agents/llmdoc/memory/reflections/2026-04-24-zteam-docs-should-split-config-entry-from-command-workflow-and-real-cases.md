# ZTeam 文档应拆开配置入口、命令工作流和真实案例

## 背景

- `ZTeam` 当前已经在 `tui` 中形成了比较稳定的本地双 worker 工作流：
  - `/zteam` 打开工作台
  - `/zteam start` 提交启动指令
  - `/zteam status` 查看状态
  - `/zteam attach` 恢复并尽量重新附着最近 worker
  - `/zteam <worker> <任务>` 分派任务
  - `/zteam relay ...` 做 worker 间消息转发
- 仓库里原有文档只在 `docs/config.md` 里留下一个简短入口说明，足够说明开关存在，但不足以支撑真实使用。

## 观察

- 像 `ZTeam` 这种“TUI 内的本地工作流”不能只写配置项。用户看到 `zteam_enabled` 后，只知道“可以开”，不知道：
  - 哪些命令是真正的主路径；
  - `/zteam start` 是提交给主线程的启动指令，不是直接绕过主线程强建 worker；
  - 运行中只允许 bare `/zteam` 和 `/zteam status`；
  - `attach`、`dispatch`、`relay` 分别解决什么问题。
- 只写命令列表也不够。没有实际使用案例时，用户很难把“工作台”“分派”“relay”“attach”串成一个完整心智模型。
- 因此这类文档要至少拆成两层：
  - `config.md`：只负责入口开关、边界和跳转；
  - `slash_commands.md`：负责命令行为、状态说明和真实协作案例。

## 结论

- 对于 `ZTeam` 这类本地模式，文档结构应遵循：
  1. 在配置文档里说明“如何启用/禁用”和“这是什么入口”；
  2. 在命令文档里写完整语法、运行中限制、工作台状态解释；
  3. 至少补 2 到 3 个真实工作流案例，覆盖：
     - 并行拆分 frontend/backend
     - 单 worker 深挖
     - 中断后 attach 恢复
- 如果一个功能既有“配置入口”又有“持续交互工作流”，就不要把全部说明压缩在同一个配置段落里。

## 后续默认做法

- 以后给 TUI 本地模式补文档时，优先检查：
  - 配置文档是否只保留开关与边界；
  - 命令文档是否完整覆盖主路径；
  - 是否有足够贴近真实工作的案例，能让用户直接照着用。
- 如果没有实际案例，说明文档仍停留在“能看懂实现”，还没到“能指导使用”。
