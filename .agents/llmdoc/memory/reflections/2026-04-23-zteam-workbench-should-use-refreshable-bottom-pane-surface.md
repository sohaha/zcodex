# ZTeam 这类持续刷新的协作工作台应落在可持续渲染的 BottomPaneView，而不是普通 history scrollback

## 背景

- `a3` 需要在 TUI 中实现一个真正可用的 ZTeam 协作工作台，让主线程持续看到 worker 状态、任务板、消息流、结果回流和阻塞提示。
- 起初一个看似便宜的方案，是沿用 history cell，把 `/zteam` 渲染成一张“入口/状态卡片”，再在后续事件里尝试更新它。

## 观察

- 普通 history scrollback 天然偏向“追加事件”，不是一个会随着共享状态持续重绘的 surface。
- 即使状态模型已经能收到 `ThreadStarted`、`ItemCompleted`、`ThreadClosed` 等通知，如果视图本身只是历史区中的一条记录，也不会稳定地跟着状态变更重绘。
- 这会把“协作工作台”退化成“某一时刻的静态快照”，用户必须依赖额外消息或重新触发命令，体验和信息密度都不对。

## 结论

- 只要目标是“持续刷新的工作台/面板/仪表盘”，优先落到可持续渲染的 surface，例如 `BottomPaneView`，并让视图直接从共享状态快照构建正文。
- history cell 更适合不可变的事件记录、一次性通知或审计轨迹；不要把它当作动态控制台来勉强复用。

## 后续默认做法

- 在 TUI 里新增类似“工作台”“监控面板”“协作看板”的功能时，先判断它是：
  - 一次性事件展示
  - 还是需要随着状态流持续更新的长期视图
- 如果是后者，默认选 `BottomPaneView` 或其他明确支持持续渲染的 surface，再把状态层做成共享快照模型；不要先把实现塞进 history，再用补丁式重绘去抢救。
