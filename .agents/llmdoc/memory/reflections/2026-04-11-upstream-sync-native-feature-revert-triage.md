# 上游同步时区分“本地分叉功能”和“上游已回滚功能”的反思

## 背景
- 这轮同步把 `openai/codex` main 合进本地 `web` 分支后，残留的主要分歧之一是 `codex-rs/account`。
- 表面上看，它像是“本地仓库里已经有的功能”，容易被误判成应该默认保留的分叉实现。

## 误判风险
- 如果只根据当前仓库是否存在代码来判断，很容易把“上游后来已经回滚的原生功能”当成本地定制功能继续保留。
- 这种误判会让同步结果悄悄偏离 upstream 的现状，而且不容易在普通冲突解决里被发现。

## 本次有效做法
- 先查提交历史确认来源，而不是只看工作树：
  - `9f2a58515` 引入了 `account` 功能。
  - `930e5adb7` 又把这项功能整体回滚。
- 确认它属于上游原生功能后，再按“是否跟随 upstream 删除”处理，而不是按“保护本地分叉功能”处理。
- 同时把同步技能更新成显式规则：本地分叉功能默认不覆盖；但上游原生功能一旦被 upstream 删除或回滚，必须提醒并询问是否跟随删除。

## 这次修正带来的收益
- 避免把已经被 upstream 放弃的能力继续带在本地分叉里。
- 把真正需要用户做的判断收敛到少数高风险决策点，而不是所有冲突都升级成询问。
- 给后续同步留下可重复使用的判定顺序，而不是继续依赖人工记忆。

## 下次怎么做
- 遇到“本地有代码、上游没了”的差异时，先查功能来源归属，再判断是保留分叉还是跟随 upstream 删除。
- 上游测试变化若只是在反映上游功能已被移除，不要自动把本地旧实现当成必须保留的既成事实。
- 把这类判断写进同步技能或 guide，避免下一次重复把“旧同步残留”误认成“本地产品决策”。
