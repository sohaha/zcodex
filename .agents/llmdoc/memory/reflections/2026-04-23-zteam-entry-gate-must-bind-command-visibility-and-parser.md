# ZTeam 入口开关必须同时绑定命令显隐与命令解析

## 背景

- `ZTeam` 的首个执行 issue 只落入口与配置，不实现实际工作台和编排。
- 该 issue 明确要求 `tui.zteam_enabled` 成为单一真相源，关闭后不能出现“入口不见了，但内部仍能走命令路径初始化”的半禁用状态。

## 观察

- 在当前 TUI 里，slash command 的“可见”与“可解析”并不是天然同一层：
  - 命令弹窗和 composer 自动补全走 `bottom_pane/slash_commands.rs` 的 gating。
  - 真正的 command dispatch 仍由 `ChatWidget` 接收 `SlashCommand` 后执行。
- 如果只在 UI 列表里隐藏命令，但不把同一布尔值接到 parser 所用的 builtins 过滤里，用户仍可能通过精确命令名走进同一路径。
- 反过来，如果只在 parser 层挡住而不改列表，用户会看到一个永远不可用的入口，形成假阳性提示。

## 结论

- 对这类本地模式入口，配置开关必须至少接到三处：
  1. 命令列表/补全显隐；
  2. 精确 slash command 解析；
  3. 实际入口 handler 的兜底检查。
- 即使第 1、2 层已经收口，第 3 层仍值得保留，原因不是做静默 fallback，而是给直接调用、未来接线变更或测试路径一个明确、可观察的禁用反馈。

## 后续默认做法

- 以后为 TUI 新增本地模式、实验工作台或专项入口时，先检查：
  - 配置是否同时驱动 popup/composer 可见性与 parser 可达性；
  - handler 是否保留显式禁用反馈；
  - 测试是否分别覆盖“可见时可进入”和“关闭时不可解析/不可进入”。
- 如果配置只是 UI 装饰开关而不是行为开关，不要沿用这种模式；但一旦文档或 issue 把它定义为“禁用该模式”，就必须把它当成行为真相源。
