# 产品需求文档 (PRD)

## 文档信息
- **功能名称**：federation-team-mode
- **版本**：1.0
- **创建日期**：2026-04-23
- **作者**：PM Agent
- **状态**：草稿

## 摘要

> 下游 Agent 请优先阅读本节，需要细节时再查阅完整文档。

- **核心目标**：把未发布的 federation 从“跨实例文本任务投递底座”重构为“主线程协调前端/后端 worker 持续协作”的 Team/Mission 模式。
- **目标用户**：独立开发者、小团队开发者、IDE/app-server 客户端用户，尤其是希望让不同 Codex 实例分别负责前端与后端并在过程中持续沟通的人。
- **关键功能**：协作 session、角色化 worker、共享任务板、worker 间中途通讯、主线程结果回流、在线状态与恢复。
- **技术约束**：不要求向后兼容现有 federation 协议与命令面；仍需保持与现有 `/root` subagent tree 语义隔离；优先复用现有 app-server `thread/start` seam 与 federation runtime。
- **优先级**：MVP 仅覆盖同机多实例、前端/后端双 worker、主线程可见协作工作流；不做跨机器联邦、自动合并、复杂群聊与权限系统。

## 1. 概述

### 1.1 背景

当前仓库里的 federation 已完成独立 `protocol / daemon / client / app-server bridge` 的最小闭环，但能力仍停留在：

- 注册实例
- 列 peer
- 发送 `TextTask`
- 收取 `TextResult`

这对“工程演示”足够，但无法支撑真实的前后端协作开发场景。用户真正想要的不是“多开几个 Codex 然后互发文本”，而是：

- 一个实例写前端
- 一个实例写后端
- 两边过程中能随时沟通
- 主线程能看见状态、阻塞和结果
- 中断后还能恢复协作关系

因此，本项目不是继续补低层 daemon 命令，而是把 federation 上升为 Team/Mission 协作工作流。

### 1.2 目标

本功能要达成的目标：

1. 提供主线程驱动的协作 session，让前端 worker 和后端 worker 成为一组可持续协作的成员。
2. 支持任务分派、中途消息、阶段结果三类核心协作事件，而不再局限于 task/result 两类原语。
3. 让主线程自然看到 worker 的在线状态、最近消息、阻塞与结果，不需要用户手工读 inbox。
4. 支持协作 session 的恢复与重连，避免 `thread/start` 一次性挂 bridge 的短生命周期限制。
5. 保持跨实例协作与现有 root tree 内多子代理语义隔离，避免污染 `/root`、`AgentPath` 与 `SessionSource::SubAgent`。

### 1.3 成功指标

- [ ] 用户可在主线程创建一个包含 `frontend` 与 `backend` 两个 worker 的协作 session。
- [ ] 两个 worker 可以在不结束当前任务的情况下互发中途消息，并被主线程看到。
- [ ] 主线程可直接看到 worker 的最新状态、最近消息和阶段结果，而不要求手动读取 daemon inbox。
- [ ] `resume` 或等效重连入口可恢复 session 关系与最近协作状态。
- [ ] 文档、协议与任务拆解足以指导后续无兼容负担的重构实现。

## 3. 功能需求

### FR-001：协作 session 创建与管理
- **描述**：当用户从主线程进入 Team/Mission 模式时，系统必须能创建一个协作 session，并把主线程、前端 worker、后端 worker 绑定到同一会话中。
- **优先级**：必须有
- **依赖**：无
- **验收标准**：
  - [ ] AC-1：系统可创建带有 `session_id`、标题、参与成员列表的协作 session。
  - [ ] AC-2：成员至少包含 `frontend`、`backend`、`coordinator` 三种角色表示方式。

### FR-002：角色化 worker 启动
- **描述**：当用户指定前端目录和后端目录时，系统必须能启动或绑定两个角色化 worker，并记录其 `role / display_name / cwd / status`。
- **优先级**：必须有
- **依赖**：FR-001
- **验收标准**：
  - [ ] AC-1：主线程可看到两个 worker 的角色与工作目录。
  - [ ] AC-2：worker 的在线状态至少区分 `online / stale / offline`。

### FR-003：任务分派
- **描述**：当主线程需要拆分工作时，系统必须能向任一 worker 发送 `task` 类型消息，并在协作 session 中记录其状态。
- **优先级**：必须有
- **依赖**：FR-001, FR-002
- **验收标准**：
  - [ ] AC-1：主线程可向前端或后端 worker 分派任务。
  - [ ] AC-2：任务在协作历史中可区分发送者、接收者、时间与状态。

### FR-004：worker 间中途通讯
- **描述**：当任一 worker 在执行中遇到字段澄清、接口变更、阻塞依赖或实现决策时，系统必须允许其向另一个 worker 发送 `message` 类型消息，而不视为任务结束。
- **优先级**：必须有
- **依赖**：FR-002
- **验收标准**：
  - [ ] AC-1：前端 worker 可直接向后端 worker 发中途消息，后端亦然。
  - [ ] AC-2：主线程能看到此类消息，并区分其不是最终交付。

### FR-005：阶段结果回流
- **描述**：当 worker 形成阶段性成果时，系统必须允许其发送 `result` 类型消息，并默认回流到主线程视图。
- **优先级**：必须有
- **依赖**：FR-003
- **验收标准**：
  - [ ] AC-1：任一 worker 的 `result` 默认回到主线程。
  - [ ] AC-2：主线程可区分结果消息与普通中途消息。

### FR-006：共享任务板
- **描述**：系统应该提供一个共享任务板，记录当前 session 内的任务、负责人、状态与阻塞关系。
- **优先级**：应该有
- **依赖**：FR-001, FR-003
- **验收标准**：
  - [ ] AC-1：任务至少支持 `pending / in_progress / blocked / completed` 四种状态。
  - [ ] AC-2：主线程能看到哪个任务归属前端或后端 worker。

### FR-007：恢复与重连
- **描述**：当主线程或 worker 重新进入会话时，系统必须能恢复协作 session、最近消息与在线状态，而不是要求用户重新建立全部关系。
- **优先级**：必须有
- **依赖**：FR-001, FR-002
- **验收标准**：
  - [ ] AC-1：session 可恢复参与成员信息与最近消息。
  - [ ] AC-2：worker 的恢复不会被误判为一个全新的团队。

### FR-008：阻塞与待决策提示
- **描述**：当 worker 表达阻塞或需要主线程决策时，系统应该在主线程显式标识待处理项。
- **优先级**：应该有
- **依赖**：FR-004
- **验收标准**：
  - [ ] AC-1：主线程能一眼识别阻塞与待决策消息。
  - [ ] AC-2：阻塞解除后状态会从主线程视图中消失或降级。

### FR-009：Mission 模式扩展位
- **描述**：系统可以为后续 Mission 模式预留 feature / milestone / validator 等概念，但在 MVP 中不要求交付完整实现。
- **优先级**：可以有
- **依赖**：FR-001
- **验收标准**：
  - [ ] AC-1：架构与协议文档中明确预留该扩展位。
  - [ ] AC-2：MVP 不因此扩大范围到复杂任务编排。

## 4. 非功能需求

### NFR-001：语义隔离
- **类型**：架构约束
- **描述**：Team/Mission 模式不得复用 `/root`、`AgentPath`、`InterAgentCommunication` 或 `SessionSource::SubAgent` 表达跨实例身份。
- **指标**：设计文档与任务拆解中无将 federated peer 映射为 `/root/...` 的方案。

### NFR-002：恢复一致性
- **类型**：可靠性
- **描述**：session 恢复后看到的成员列表、最近消息与在线状态必须来自同一套协作真相源。
- **指标**：恢复链路文档中不出现“双份真相源”。

### NFR-003：可观察性
- **类型**：可运维性
- **描述**：主线程必须能观察每个 worker 的在线状态、最后活跃时间、最近消息和任务状态。
- **指标**：UI/协议至少暴露 `status / lastHeartbeatAt / lastMessageAt`。

### NFR-004：低认知负担
- **类型**：可用性
- **描述**：普通用户不应直接面对 `instance_id`、raw inbox、ack 文件等底层概念。
- **指标**：产品工作流以 session、role、message、task、result 为主词汇。

### NFR-005：同机优先
- **类型**：范围控制
- **描述**：MVP 仅面向同机多实例，不要求跨机器网络协议、鉴权与远程路由。
- **指标**：架构与任务不包含跨机器 transport 设计。

### NFR-006：无兼容包袱
- **类型**：交付策略
- **描述**：由于 federation 尚未发布，可直接重构，不要求兼容当前外露命令面或协议字段。
- **指标**：设计与任务明确以“直接重构”为原则，而非过度迁移兼容。

## 5. 用户故事

### US-001：作为主线程协调者，我想创建前端/后端双 worker 协作会话，以便同时推进两部分工作。
- **作为** 主线程协调者
- **我想要** 一键建立前端 worker 与后端 worker 的协作 session
- **以便** 不必在不同实例间手工维护关系
- **验收标准**：
  - [ ] 能看到两个 worker 的角色、目录和在线状态
  - [ ] 能向任一 worker 分派首个任务

### US-002：作为前端 worker，我想在开发中随时向后端 worker 询问接口问题，以便减少等待和重复解释。
- **作为** 前端 worker
- **我想要** 发送中途消息而不结束当前任务
- **以便** 快速确认字段、状态或 API 变化
- **验收标准**：
  - [ ] 消息不会被系统当作最终结果
  - [ ] 主线程也能看到这次沟通

### US-003：作为后端 worker，我想把阶段性交付回流给主线程，以便主线程决定是否继续推进或集成。
- **作为** 后端 worker
- **我想要** 发送 `result` 类型消息
- **以便** 主线程看到阶段性成果和下一步建议
- **验收标准**：
  - [ ] 结果在主线程中被标记为交付
  - [ ] 结果与普通 message 在视觉与语义上区分

### US-004：作为主线程协调者，我想看到当前任务板和阻塞项，以便及时介入决策。
- **作为** 主线程协调者
- **我想要** 查看任务板与阻塞状态
- **以便** 在合适的时候介入，而不是盲等
- **验收标准**：
  - [ ] 每个任务可看到 owner 与状态
  - [ ] 阻塞会被显式提醒

### US-005：作为恢复后的用户，我想重新进入上一次的协作 session，以便继续之前的工作而不是重建团队。
- **作为** 重新进入系统的用户
- **我想要** 恢复 session、消息和状态
- **以便** 不中断连续的协作过程
- **验收标准**：
  - [ ] 最近消息可见
  - [ ] 成员关系与角色不丢失

## 6. 范围定义

### 6.1 范围内
- 同机多实例前后端双 worker 协作。
- 主线程协调视角。
- 协作 session、角色化 worker、任务/message/result 三类核心消息。
- 主线程结果回流与在线状态展示。
- 基础共享任务板。
- 恢复与重连语义。

### 6.2 范围外
- 跨机器 federation。
- 自动冲突合并与自动代码整合。
- 多于两个 worker 的复杂网状编排。
- 富媒体附件、结构化 diff 共享、语音协作。
- 完整项目管理系统、复杂权限与组织级治理。

## 7. 风险与依赖

### 7.1 风险
| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 将 federated peer 错误复用为 `/root` tree 成员 | 中 | 高 | 在架构与任务中显式禁止，保持 `InstanceId` 与 `AgentPath` 分离 |
| 只扩协议不补主线程视图，导致产品仍不可用 | 高 | 高 | MVP 优先交付协作工作台和结果回流 |

| app-server `thread/start` 接入 seam | 技术 | 就绪 | 后端 |
| TUI/app-server 客户端工作台视图 | 技术 | 待实现 | 前端 |
| Team/Mission 模式术语与用户工作流统一 | 产品 | 待定稿 | PM / 架构 |
