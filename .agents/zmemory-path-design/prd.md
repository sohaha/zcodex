# 产品需求文档 (PRD)

## 文档信息
- **功能名称**：zmemory-path-design

- **创建日期**：2026-03-30

## 摘要

> 下游 Agent 请优先阅读本节，需要细节时再查阅完整文档。

- **核心目标**：通过只暴露单一 `zmemory_path` 配置，使 zmemory 默认按项目隔离数据库，并在需要时允许开发者指定任意路径。
- **目标用户**：需要在多个 `codex-cli` 仓库或非 git 目录间切换的开发者及自动化脚本，避免全局单库带来的数据干扰。
- **关键功能**：支持 `zmemory_path` 的绝对路径与相对路径（优先相对于主仓库根，否则相对 `cwd`）；未配置时在主仓库根或当前目录下生成稳定隔离目录；自动生成目录使用稳定 key/hash，而非直接拼接源路径；通过现有 `stats/doctor` 输出解析结果。
- **技术约束**：不得引入 `scope`；保持与现有 `codex_home` 存储兼容；路径生成逻辑要跨平台一致，可预测并日后可插拔不同隔离策略。
- **优先级**：MVP（必须交付）：先完成配置与默认隔离行为，后续再扩展多 DB 管理或 GUI。

## 1. 概述

### 1.1 背景
当前 `codex-zmemory` 构建数据库路径固定在 `CODEX_HOME/zmemory/zmemory.db`（见 `zmemory/src/config.rs` 的 `zmemory_db_path`），导致所有仓库/目录共享同一个数据库。跨多个仓库、跨子路径或在非 git 目录运行时，数据混杂、权限冲突和调试困难，且缺少可观察的路径上下文。

### 1.2 目标
新增 `zmemory_path` 配置作为唯一对外入口，使开发者可覆盖数据库位置。未配置时，自动在主仓库根或当前目录下按项目隔离、生成稳定目录路径（基于 canonical repo/cwd 的 key/hash），确保同一工作空间每次启动都能找到同一个数据库同时避免与其他 workspace 交叉。

### 1.3 成功指标
- [ ] Git 工作空间默认写入独立、稳定目录；同一主仓库下的 worktree 继承同一个默认数据库身份，不与其他仓库混用。
- [ ] `zmemory_path` 支持绝对/相对输入，相对路径在 git 内相对于主仓库根，否则相对于 `cwd`，最终解析路径可在日志和现有诊断输出里查看。
- [ ] 默认隔离目录使用可复现的 key/hash，移动仓库或子目录时只要 canonical 路径一致即可复用，默认路径不包含用户原始目录名。

## 2. 用户画像

### 用户画像 1：跨仓库内存调试者
| 属性 | 描述 |
|------|------|
| 角色 | 负责 Codex CLI 多项目开发与自动化脚本的工程师 |
| 特征 | 同时打开多个 `codex-cli` 仓库、会在不同 subdir 启动 CLI；部分项目仍使用本地目录而非 git；对记忆历史高度敏感 |
| 需求 | 每个仓库/目录都能独立维护 zmemory 数据，不希望切换目录带来旧数据干扰，且要知道实际数据库在哪 |
| 痛点 | 目前各项目共用一个数据库，读写互相影响，无法只清理某个项目的记忆；CI/脚本在非 git 目录时不知道会读哪一个 DB |
| 期望 | 有单一配置可控制路径；默认隔离满足大部分 repo，且可以在日志或 CLI 输出中确认最终路径 |

## 3. 功能需求

### FR-001：暴露 `zmemory_path` 配置
- **描述**：为 zmemory 只新增一个配置 `zmemory_path`（通过现有配置加载链读取），支持绝对路径直接使用，相对路径按主仓库根或 `cwd` 解析，最终路径在启动日志和现有诊断输出中明确可见，方便调试。
- **优先级**：必须有
- **依赖**：配置加载器（codex-cli config）、git 仓库根探测逻辑
- **验收标准**：
  - [ ] AC-1：设置绝对路径时，数据库直接写入该路径，并可通过 `codex zmemory doctor`/`stats` 或日志确认。
  - [ ] AC-2：设置相对路径时，在 git 仓库内相对于主仓库根，否则相对于 `cwd`；同一主仓库下从不同子目录或 worktree 启动时，若主仓库根一致则返回同一路径。
  - [ ] AC-3：配置文档/README 明确描述绝对/相对/默认行为，包含示例。
  - [ ] AC-4：CLI 启动成功后能记录/打印解析结果，且 `doctor`/`stats` 可让开发者确认所在路径。
- **边界情况**：
  - 相对路径中含 `..` 或路径不存在时，报错并指导用户使用绝对路径或提前创建目录。
  - 提供的路径指向目录时，在其下创建 `zmemory.db` 不影响其他文件。

### FR-002：不配置时按项目隔离数据库
- **描述**：`zmemory_path` 未设置时，根据当前工作目录是否在 git 仓库决定隔离策略，在 git 内以主仓库根为 anchor，非 git 目录以 `cwd` 为 anchor，生成目录时不直接拼源路径而是用 canonical 路径 + 稳定 key/hash。
- **优先级**：必须有
- **依赖**：git 识别逻辑、路径 canonical 化工具、`codex_home` 仍作为根存储
- **验收标准**：
  - [ ] AC-1：在 git 项目中，生成目录路径形如 `{codex_home}/zmemory/workspace-<repo_hash>/zmemory.db`，hash 基于 canonical 主仓库根且跨平台一致。
  - [ ] AC-2：在非 git 目录（例如 CI 临时目录），使用 `{codex_home}/zmemory/workspace-<cwd_hash>/zmemory.db`，hash 保证与当前目录对应且移动后仍稳定。
  - [ ] AC-3：默认目录由稳定 key/hash 构造，不直接包含原始 repo/cwd 名称，便于今后重命名或移动目录时仍可复用。
- **边界情况**：
  - git 无法识别时（没有 git，裸目录），走 `cwd` hash。
  - canonical 路径发生变化（例如符号链接），hash 采用 realpath 版本，减少无谓迁移。

### FR-003：日志/状态中可观察路径解析过程
- **描述**：默认路径解析需要具备可观测性，执行解析后将结果附加到 zmemory 启动日志以及现有 `codex zmemory doctor`/`stats` 输出，便于定位在某个目录运行时实际访问的是哪个 DB；首版不新增额外 CLI 命令或命令行 flag。
- **优先级**：应该有
- **依赖**：日志体系或 CLI 诊断命令
- **验收标准**：
  - [ ] AC-1：启动时打印 `Resolved zmemory path: ...`（或等效）的信息。
  - [ ] AC-2：现有诊断命令能返回当前 `zmemory_path`（优先列在 `doctor`/`stats` 输出，不要求首版新增专用命令）。
  - [ ] AC-3：当解析失败（例如权限不足）时，明确给出原因并提示用户先设置合适路径。
- **边界情况**：
  - 非 git 环境不能默认打印 git 信息，只需说明使用 `cwd`。
  - 日志级别可调（info/debug），避免误爆生产环境日志。

## 4. 非功能需求

### NFR-001：兼容性与可预测
- **类型**：兼容/可预测
- **描述**：保持旧全局路径 `codex_home/zmemory/zmemory.db` 可通过显式 `zmemory_path` 继续访问；git/非 git 情况下默认路径 deterministic，hash 计算在各平台一致。
- **指标**：同一路径多次启动得到同一 hash，`codex_home` 下结构没有重复目录；已有数据库文件如需复用，必须通过显式 `zmemory_path` 指向旧路径。

### NFR-002：可观测与可调试
- **类型**：可观测
- **描述**：路径解析过程及最终 `zmemory.db` 位置必须记录在日志或诊断输出中，便于用户在 multi-repo/CI 场景下判断当前写入哪个数据库。
- **指标**：`codex zmemory doctor` 或 `codex zmemory stats` 输出含 `current_zmemory_path`/`pathResolution` 字段；日志中的 `Resolved zmemory path` 最多 1 条/启动。

### NFR-003：跨平台
- **类型**：跨平台
- **描述**：路径解析逻辑在 Linux/macOS/Windows 下行为一致，hash 输入使用 canonical `PathBuf` 与统一哈希算法（如 SHA256），不依赖额外 lowercase/casefold 折叠。
- **指标**：在三类系统中，对相同 repo/cwd（在 git 中 canonical 后相同）产出相同 hash，并通过单元测试覆盖。

## 5. 用户故事

### US-001：多个仓库隔离记忆
- **作为** 多仓库开发者
- **我想要** 每个仓库都写入独立的 zmemory 数据库
- **以便** 切换仓库时不会读取或污染其他项目的记忆
- **验收标准**：
  - [ ] 在 repo A 和 repo B 中启动 `codex zmemory` 时路径不同，数据库互不影响
  - [ ] 不配置 `zmemory_path` 时自动生成的路径包含 workspace hash，且在相同主仓库下多次启动路径一致
- **优先级**：P0

### US-002：非 git 临时目录能稳定隔离
- **作为** 运行在临时目录的自动化脚本
- **我想要** 即使目录没有 git 元信息，也能拥有专属 zmemory 数据库
- **以便** CI 或临时调试不会误操作其他环境的记忆
- **验收标准**：
  - [ ] 在无 git 信息的目录启动后生成独立 hash 路径
  - [ ] 目录移动后，新的 canonical `cwd` 会映射到新的默认路径；若需复用旧数据，用户可显式设置 `zmemory_path`
- **优先级**：P1

## 6. 范围定义

### 6.1 范围内
- 提供唯一 `zmemory_path` 配置，支持绝对/相对路径及默认行为。
- 对 git 仓库和普通目录分别生成隔离数据库，生成逻辑基于 canonical path + stable key/hash。
- 将路径解析结果可观测，补充文档说明新配置与默认行为。

### 6.2 范围外
- 不新增 `scope` 概念或多路径配置。
- 不做 Web/UI 改动，仅关注 CLI/服务端逻辑与文档。
- 不自动迁移旧 `zmemory.db` 到新 hash 目录，迁移由手动配置 `zmemory_path` 驱动。

## 7. 风险与依赖

### 7.1 风险
| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| git root 探测失败（无 git、`git` 不在 PATH） | 中 | 中 | 明确使用 `cwd` hash，并在日志提示缺失 git；增加单元测试模拟 |
| hash 算法的 canonical 处理不一致导致多个 hash 目录 | 低 | 高 | 使用 realpath/canonicalize 后的原始规范路径做哈希输入，不额外 lowercase/casefold；增加跨平台单元测试 |
| 新目录无写权限导致 `zmemory.db` 无法创建 | 中 | 高 | 解析后先检查权限，快速失败并提示 `zmemory_path` 可替代 |
