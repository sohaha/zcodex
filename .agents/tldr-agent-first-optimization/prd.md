# 产品需求文档 (PRD)

## 文档信息
- **功能名称**：tldr-agent-first-optimization
- **版本**：2.0
- **创建日期**：2026-03-30
- **作者**：PM Agent
- **状态**：已批准

## 摘要

> 下游 Agent 请优先阅读本节，需要细节时再查阅完整文档。

- **核心目标**：把内嵌 `tldr` 设计成 Codex 的代码理解默认底座，而不是一个“可选高级工具”。
- **目标用户**：在大型代码库中工作的 Codex / agent 用户，以及维护 Codex 路由、提示词和工具运行时的工程师。
- **关键能力**：原生拦截与重写、结构化问题优先 `tldr`、混合问题先结构后验真、显式 raw 逃生路径、可观测的降级与失败语义。
- **产品定位**：不同于外接 hook 方案，本方案依赖内嵌 runtime、tool router、handler 与上下文状态，直接做系统级重写。
- **交付策略**：一次性按最终架构设计，允许分阶段落地，但禁止把最终方向锁死在保守 MVP 上。

## 1. 概述

### 1.1 背景
当前仓库已经把 `tldr` 内嵌到 core、cli 与 mcp 相关链路中，因此能力边界明显大于外接型方案：

1. 我们可以直接在 runtime 内部识别问题类型，而不只是靠外部 hook 拦截命令。
2. 我们可以对工具调用做原生改写、上下文注入和状态记忆，而不是只能阻断后提示用户换命令。
3. 我们可以把 `tldr` 的结构化结果与 `AutoTldrContext`、tool routing、degraded/structured failure 统一起来。
4. 因此最终目标不应只是“鼓励 agent 多用 tldr”，而应是“让系统默认先经过 tldr，再决定是否读原始文件或跑原始搜索”。

### 1.2 产品目标
- 让结构化代码问题默认先走 `tldr`。
- 让混合问题默认采用“先结构、后验真”的两阶段工作流。
- 让 shell / grep / read / task planning 都能共享同一套 `tldr-first` 决策规则。
- 保留显式 raw grep/read、regex、逐字阅读等逃生路径，并保证这些路径优先级明确。
- 让 `tldr` 的失败、降级、daemon 不可用等状态可见、可解释、可恢复。

### 1.3 成功指标
- [ ] 结构化代码问题在默认情况下优先命中 `tldr`，而不是先广泛 grep/read。
- [ ] 混合问题默认先 `tldr` 建结构，再读精确源码验证默认值、Cargo feature、README、测试等事实。
- [ ] shell 搜索、`grep_files`、`read_file`、task planning 共享统一路由语义，而不是各自为政。
- [ ] 用户显式要求 raw grep/read 时，不发生隐式偷换。
- [ ] 降级与失败状态通过 `degradedMode` / `structuredFailure` 向 agent 暴露，不伪装为正常成功。

## 2. 用户画像

### 用户画像 1：结构化分析开发者
| 属性 | 描述 |
|------|------|
| 角色 | 使用 Codex 分析大型仓库的开发者 |
| 特征 | 常问“定义在哪、谁调用它、改动影响什么、相似实现在哪” |
| 需求 | 快速拿到结构化上下文而不是海量原始文本 |
| 痛点 | agent 容易先广泛读文件和 grep，路径不稳定 |
| 期望 | 系统默认先用 `tldr`，只在必要时再读原始源码 |

### 用户画像 2：事实核对型开发者
| 属性 | 描述 |
|------|------|
| 角色 | 需要确认配置默认值、feature gate、README、测试覆盖的开发者 |
| 特征 | 问题精确、需要可引用事实 |
| 需求 | 明确区分“结构分析结果”和“源码事实结论” |
| 痛点 | 结构分析工具可能被误用为事实来源 |
| 期望 | 系统先用 `tldr` 缩小范围，但最终仍回到源码/测试核对 |

### 用户画像 3：运行时维护者
| 属性 | 描述 |
|------|------|
| 角色 | 维护 Codex router、tool spec、handler、session context 的工程师 |
| 特征 | 更关心默认策略、可解释性、可观测性和演进成本 |
| 需求 | 一套统一、可测试、可扩展的内嵌 `tldr` 框架 |
| 痛点 | 只靠提示词不稳定，只靠局部重写不完整 |
| 期望 | 形成从 prompt 到 router 到 handler 到 session 的完整闭环 |

## 3. 功能需求

### FR-001：统一的问题分类与决策模型
- **描述**：系统应先把代码库问题分为结构化问题、纯事实核对问题、混合问题三类，再决定是否先用 `tldr`。
- **优先级**：必须有
- **依赖**：无
- **验收标准**：
  - [ ] AC-1：文档和工具描述中都明确三类问题的默认工作流。
  - [ ] AC-2：结构化问题默认 `tldr-first`。
  - [ ] AC-3：混合问题默认“先 `tldr`，再读精确源码”。

### FR-002：原生工具拦截与重写层
- **描述**：系统应像 RTK 一样对相关工具与命令形态进行内嵌拦截/重写，而不是依赖外接 hook。
- **优先级**：必须有
- **依赖**：FR-001
- **验收标准**：
  - [ ] AC-1：`grep_files` 继续支持 `tldr` 自动改写。
  - [ ] AC-2：为 `read_file` 设计结构化问题前置 `tldr` 的重写/前置分析路径。
  - [ ] AC-3：为 shell 搜索类命令设计识别与软/硬重写能力。
  - [ ] AC-4：显式 raw grep/read 指令拥有高优先级逃生路径。

### FR-003：统一的 `tldr-first` 提示词与工具描述
- **描述**：tool description、agent-first 文档和运行时默认提示应统一表达同一套 `tldr-first` 规则。
- **优先级**：必须有
- **依赖**：FR-001
- **验收标准**：
  - [ ] AC-1：`tldr` tool description 明确结构化优先、事实核对需源码验证、混合问题先结构后验真。
  - [ ] AC-2：agent-first 指南补齐起手式和绕路回退规则。
  - [ ] AC-3：文案与实现不存在冲突承诺。

### FR-004：正向与负向路由指令并存
- **描述**：系统既要能识别“先用 `tldr`”，也要能识别“不要 `tldr` / raw grep / regex / 逐字阅读”。
- **优先级**：必须有
- **依赖**：FR-002
- **验收标准**：
  - [ ] AC-1：存在正向 `force_tldr` 指令。
  - [ ] AC-2：存在负向 `disable_auto_tldr_once`、`force_raw_grep`、`force_raw_read`。
  - [ ] AC-3：冲突时 raw/explicit 禁用优先。

### FR-005：上下文记忆与跨工具复用
- **描述**：系统应把最近一次成功 `tldr` 分析的项目、语言、符号、查询、路径作为统一状态，供后续重写与规划复用。
- **优先级**：必须有
- **依赖**：FR-002
- **验收标准**：
  - [ ] AC-1：`AutoTldrContext` 持有足够的上次成功信息。
  - [ ] AC-2：显式 `tldr` 请求下 safe 模式可复用最近语言。
  - [ ] AC-3：后续 `read_file` / shell / planning 可以复用同一状态。

### FR-006：可观测降级与失败语义
- **描述**：当 daemon 不可用、回退到 local engine、或只能拿到诊断信息时，系统必须显式暴露降级状态。
- **优先级**：必须有
- **依赖**：FR-002
- **验收标准**：
  - [ ] AC-1：`degradedMode` 与 `structuredFailure` 被稳定透出。
  - [ ] AC-2：agent 不把降级结果当作“正常成功”。
  - [ ] AC-3：daemon-only action 与可本地回退 action 行为边界清晰。

### FR-007：Session 级 warm / preload 策略
- **描述**：系统应在合适的阶段预热 `tldr` daemon 或索引，降低首次使用延迟。
- **优先级**：应该有
- **依赖**：FR-005
- **验收标准**：
  - [ ] AC-1：存在 session-start 或首次结构化问题触发的 warm 策略。
  - [ ] AC-2：warm 不影响 raw 路径和非代码问题。
  - [ ] AC-3：warm 状态可观测。

## 4. 非功能需求

### NFR-001：正确性优先
- **类型**：正确性
- **描述**：结构化分析与源码事实必须分层；不得用 `tldr` 替代需要精确核对的源码事实。
- **指标**：默认值、feature gate、README、测试覆盖相关问题必须保留源码核对路径。

### NFR-002：最小惊讶原则
- **类型**：体验
- **描述**：用户显式要求 raw grep/read/regex 时，系统不应暗中改写成 `tldr`。
- **指标**：冲突指令优先级有明确规则和测试。

### NFR-003：可解释性
- **类型**：可维护性
- **描述**：agent 为什么走 `tldr`、为什么没有走、为什么降级，都应从文档、日志或 payload 中解释清楚。
- **指标**：关键路径有文档、路由 reason 或降级字段支撑。

### NFR-004：渐进落地但终局一致
- **类型**：架构
- **描述**：允许分阶段实施，但每个阶段都必须朝最终内嵌 `tldr` 重写层收敛，而不是引入未来要推倒的临时方案。
- **指标**：各阶段任务文档明确对应最终架构位置。

## 5. 用户故事

### US-001：结构化代码问题自动进入 `tldr-first`
- **作为** 提问“谁调用它 / 改动影响什么 / 相似实现在哪”的开发者
- **我想要** 系统默认先用 `tldr`
- **以便** 更快进入结构化理解，而不是先大范围 grep/read
- **验收标准**：
  - [ ] 结构化问题默认先走 `tldr`
  - [ ] raw grep/read 只在显式请求时优先

### US-002：混合问题先结构再核对事实
- **作为** 既想理解结构、又想确认默认值/feature/README 的开发者
- **我想要** 系统先用 `tldr` 缩小范围，再自动回到源码验证
- **以便** 同时获得高效率与高准确性
- **验收标准**：
  - [ ] 混合问题在文档和运行时都有对应工作流
  - [ ] 最终回答能区分结构结论与事实结论

### US-003：用户显式控制路由
- **作为** 熟悉工具行为的高级用户
- **我想要** 说“先用 `tldr`”时系统更积极，说“不要 `tldr` / raw grep”时系统严格服从
- **以便** 能控制分析路径
- **验收标准**：
  - [ ] `force_tldr` 与 raw/disable 指令都生效
  - [ ] 冲突优先级稳定且可测试

## 6. 范围定义

### 6.1 最终版本范围内
- 统一的问题分类模型
- `tldr` tool description / agent-first 指南 / 默认提示统一化
- `grep_files`、`read_file`、shell 搜索类命令、task planning 的原生拦截与重写设计
- `AutoTldrContext` 的跨工具复用
- 显式 raw / regex / verbatim 逃生路径
- daemon-first、local fallback、diagnostic-only 的统一语义处理
- session warm / preload 策略
- 完整测试与可观测性设计

### 6.2 不在范围内
- 把 `tldr` 当成唯一搜索工具，完全删除 raw grep/read
- 为纯聊天、纯写作、非代码问题强行引入 `tldr`
- 用隐藏 fallback 掩盖失败

## 7. 风险与依赖

### 7.1 风险
| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| shell 层重写过激导致误判 | 中 | 高 | 先做软拦截/提示，再逐步收紧，保留 raw 逃生路径 |
| 结构分析和事实核对边界模糊 | 中 | 高 | 在文档与实现中强制区分“结构结果”与“源码事实” |
| 内嵌策略过多导致维护复杂 | 中 | 中 | 所有重写入口都共享统一的分类与指令模型 |
| warm / preload 增加启动成本 | 低 | 中 | 采用惰性 warm 或条件触发策略 |

### 7.2 依赖
| 依赖项 | 类型 | 状态 | 负责人 |
|--------|------|------|--------|
| `codex-rs/core` 的 tool routing 与 handler 链路 | 技术 | 就绪 | Core |
| `codex-native-tldr` 的 daemon / fallback / wire contract | 技术 | 就绪 | Native TLDR |
| `AutoTldrContext` 与 turn/session 上下文状态 | 技术 | 就绪 | Core |
| 日志与测试基础设施 | 技术 | 就绪 | Core |
